# ImageForge（仓颉 / HarmonyOS）

ImageForge 是基于**仓颉（Cangjie）+ ArkUI 声明式 UI** 的 HarmonyOS 原生应用，通过 HTTP 对接 PTP 网关（默认 `https://ptp.matrixlabs.cn`），服务端再转发到 ComfyUI 工作流完成**图生图编辑**与**文生图**。工程为单模块 `entry`，UI 与业务逻辑几乎全部由 `.cj` 实现，无业务向 ArkTS/ETS 分流。

---

## 一、项目架构（通读后的整体理解）

### 1.1 分层与职责

| 层次 | 路径 / 组件 | 职责 |
|------|-------------|------|
| **宿主与窗口** | `main_ability.cj` | `UIAbility`：全屏布局、读取系统安全区（顶部状态栏/底部导航条）宽高写入 `AppStorage`（`topHeight` / `bottomHeight`），写入 `forgeFilesDir` 供创作页保存/预览落盘。 |
| **阶段入口** | `ability_stage.cj` | `AbilityStage` 生命周期与日志。 |
| **壳层 / 路由** | `index.cj`（`@Entry` → `EntryView`） | **未登录**：登录页；**已登录**：主页 `Column` + **顶部悬浮用户信息条** + **中间 Swiper 四 Tab**（首页/创作/历史/设置）+ **底部 Tab 导航**。集中管理 `accessCode`、积分、语言、`editorMainMode` / `editorEditWorkflow`、各页安全区内边距 Props。 |
| **功能页** | `home_page.cj` | 品牌区、四宫格入口、快速开始；滚动时玻璃/模糊与透明度随 `scrollY` 变化。 |
| | `editor_page.cj` | **创作**：`Stack` 全屏 Scroll + **悬浮模式切换条**（编辑/生成）；编辑支持演示图/URL；生成走 `apiGenerateImage`，成功后尽力 **下载落盘 + `file://` 预览**（失败则回退 HTTPS URL）。 |
| | `history_page.cj` | 读 `AppStorage` 键 `imageForgeHistoryLines`（`@StorageProp`），左图右文列表，点击全屏预览；支持「刷新」与新旧两种历史行解析。 |
| | `settings_page.cj` | 健康检查、版本信息、说明文案等（相对其它页迭代较少）。 |
| **服务与模型** | `api_service.cj` | HTTP 客户端封装、JSON 轻量解析、`UserInfo` / `ImageTaskResult` 等类型、**结构化历史写入**（`\u001E` 分隔字段 + 时间戳）、与 PTP API 对齐的路径与超时。 |

此外，`entry/src/main/cangjie/` 下另有体积很小的 **`ability_mainability_entry.cj`、`module_entry_entry.cj`**（多为工程/入口桩或生成相关保留文件），与主业务能力弱相关，但在统计代码量时仍会计入同一目录。

### 1.2 数据流与状态

- **会话态**：登录态、`accessCode`、用户名、积分、当前 Tab、创作子模式等均位于 `EntryView` 的 `@State`，向下通过 **`@Prop` 传递到子页面**。
- **跨页共享**：安全区、`forgeFilesDir`、历史文本 blob 使用 **`AppStorage`**，避免 Ability 与子组件层层传参。
- **历史记录**：完成任务时由 `editor_page` 调用 `imageForgeHistoryAppendRecord`：**时间 + 卡片标题 + 提示词摘要 + 图片 URL**（不截断 URL）；`history_page` 解析后与旧版「中文 · 分隔」行兼容。
- **网络**：`module.json5` 声明 `ohos.permission.INTERNET`；编辑为多部分表单上传字节；生成以 **非流式** `POST /api/generate` 为主路径（相对稳定解析结果），必要时仍可保留流式能力在 `api_service` 内供演进。

### 1.3 与后端的契约

服务端行为以仓库内 **`PTP/API_DOCUMENTATION.md`** 为准。客户端将相对路径的图片地址用 `API_BASE_URL` 拼成可请求的 URL。

### 1.4 构建与模块边界

- 构建链路：**Hvigor + cjpm**（参见 `.agents/skills/harmonyos-build`）。
- 业务代码边界清晰：**界面在各自 `*_page.cj`，网络与历史在 `api_service.cj`，壳逻辑在 `index.cj`**。当前主要「体积压力」集中在 `api_service.cj` 与 `editor_page.cj`。

## 三、代码量门禁（`.`cj`）与合规结论

**统计口径说明**

- **A（建议用于「业务体量」对标）**：`entry/src/main/cangjie/**/*.cj`——工程主源码目录。
- **B（更宽口径）**：`entry` 下所有路径中包含 `cangjie` 目录的 `.cj`（含单元测试、`ohosTest` 等）。

**实测（约稿时脚本统计 UTF-8 字符数、按换行拆分逻辑行）**

| 口径 | 文件数 | 逻辑行数 | 字符数 |
|------|--------|----------|--------|
| A `entry/src/main/cangjie` | 10 | **约 3652** | **约 117557** |
| B `entry/.../cangjie/*.cj` | 15（含测试树） | 约 3723 | 约 118994 |

**与题目约束对照**

| 约束 | A 口径结论 |
|------|------------|
| 逻辑行 ≤ 2000 | **不达标**（约 3652 行）。 |
| 字符数 ≤ 120000 | **达标**（约 117557，尚余约 2.4K 字符余量）。 |
| B 口径字符数 | 约 118994，仍 **低于 120000**；但若继续膨胀易触碰上限。 |

> 以上为仓库当前状态的静态统计；重修命名或格式化会轻微波动属正常。

### 3.1 超行数原因（不写代码的结论性研判）

1. **`api_service.cj` 单体过大**：同时承担 HTTP、multipart、JSON 容错解析、多业务实体、历史序列化，`import` 与符号面都宽，线性增长最快。  
2. **`editor_page.cj` UI 密度高**：单文件内并排维护「编辑 + 生成」两套表单、多套 `@Builder`，装饰链长、状态字段多。  
3. **`index.cj` 承担壳层过多**：登录 + 主页布局 + Tab + 与各子页耦合的 spacer 计算，易产生长文件。  
4. **重复的「设计 token」未抽象**：渐变按钮、玻璃卡片边距若在多处手写，会形成大量视觉相关行数却无业务增量。

### 3.2 优化建议清单（仅建议，本条回复未改业务代码）

**若目标是压到 ≤2000 行（需砍掉约 40%+ 体量，难度高，建议分阶段）：**

| 优先级 | 方向 | 说明 |
|--------|------|------|
| P0 | **拆分 `api_service.cj`** | 例如：`api_http.cj`（request/销毁/主线程派发）、`api_models.cj`（数据类）、`api_json.cj`（apiJson\*）、`history_store.cj`（与 AppStorage/history 相关）。 |
| P0 | **拆分 `editor_page.cj`** | `editor_edit_panel.cj` / `editor_generate_panel.cj` / `editor_shared_styles.cj`（或 `@Builder` 聚合），页面文件只拼装。 |
| P1 | **`index.cj` 瘦身** | `login_shell.cj`、`main_tabs.cj`、`nav_bar_builders.cj` 将登录与 Tab 拆开，壳文件只保留状态与拼装。 |
| P1 | **样式与文案收敛** | 颜色、圆角、统一字号放入少量常量函数或最小「主题」结构，删掉重复字面量链。 |
| P2 | **评估 stub 文件** | `ability_mainability_entry.cj`、`module_entry_entry.cj` 若仅为模板残留且与编译无关，可考虑移除或并与文档说明（需你本地构建验证）。 |
| P2 | **JSON 解析** | 长期可评估是否只允许服务端返回严格 JSON，从而收缩手写扫描逻辑（需服务端配合）。 |

**若目标仅为「卡在 120000 字符内」**：当前 A 口径仍有余量；优先控制 **`api_service` 与长篇字符串/注释**、避免再在单文件堆放新功能。

## 四、ArkTS / TypeScript 占比（15% 规则）

**规则理解**：允许使用不超过「建议最高字符数」**120000 的 15%**——即 ArkTS（及题目认可的 TS 混入）折算约 **≤ 18000 字符**，且通常仍「计入总量」扣分逻辑需按课程说明最终解释。

**本仓库实测（自编工程文件，不含 `node_modules`、`oh_modules` 依赖源码）**

- `entry/src` 下：**无自编 `.ets` 业务源码**。
- **根目录 / `entry/hvigorfile.ts`**：两处 `hvigorfile.ts` 合计约 **454 字符**（构建脚手架，非 ArkUI 业务）。

**合规结论（按常见理解）**

- 自编 ArkTS **远低于** 18000 字符上限；相对 **≈117557 字符的仓颉主源码**，占比 **远低于 15%**。  
- 若评课方将 **`oh_modules` 第三方 `.ets` 或构建产物 `BuildProfile.ets`** 一并计入：总量会上升，但那属于依赖/生成物，建议在作业说明中单列「不计入自编」或通过 `.gitattributes`/统计脚本剔除后再报数。

**若不合规怎么办（不改代码的规划）**：将业务继续放在 `.cj`；仅保留极小 `hvigorfile.ts`；避免新增 `Ability`/`页面` 的 `.ets` 桥接层。

## 六、你的产品思考：逐项评判与是否写进路线图

以下内容对你列出的 **0～6** 做**利弊与可行性**简评（均已吸收进迭代建议）。

### 0. 登录界面改成蓝色、风格一致

- **赞成**：与应用内主色 `#007AFF` 体系统一，品牌化更强。  
- **注意**：全蓝底需控制**对比度**（输入框与错误提示可读性）；可保留浅色卡片承载表单，仅用蓝做头图/按钮/强调条，免得「整块蓝屏」压低质感。

### 1. 底部导航与内容区间距大、遮盖多 → 悬浮式

- **赞成**：你与顶部悬浮栏对齐的思路合理；底部「大块占位」往往来自 **`bottomHeight` padding + Tab 自建高度 + Scroll bottom padding** 的叠加，改成 **浮在内容之上**（或缩小 tab 高度/`padding`）能显著减负。  
- **实现要点**：需在 `index.cj` 中把 Tab 从「占 layout 尾的 Column」改为 **`Stack` 底部对齐 + 透明点击区**，并重新计算 **`bottomBarOverlapVp` / Scroll 文末留白**，避免裁剪或误触。

### 2. 首页字小、整体上偏 → 内容与底部留白

- **赞成**：视觉重心偏上会与「底部占位」相加显得「下腹空」。  
- **可做**：适度增大卡片标题字号、行间距；在中间增加**轻量级信息模块**（如「最近一条历史」摘要、或服务状态单行）填满垂直节奏，而不是单纯放大空白。

### 3. 创作-编辑：传图困难；交互两套逻辑；排版缺层次

- **传图**：当前路径不是系统相册，`INTERNET + URL` 在真机上受网络/Cookie/服务端限制大；**与高完成度竞品对比时最吃亏**。参考工程已演示 **PhotoAccessHelper** 路线——这是「能力级」差别，不光是 UI。  
- **交互统一**：Workflow 三段与预设按钮的**选中态样式**不一致，违反 **一致性启发式**，用户会认为「不是一种控件」。建议统一为：**同一套 Capsule/Text Chip** 的 `@State selectedId` + 同一套边框/底色规则。  
- **分段标题**：你给「模块化小标题」的方向正确，能降低扫描成本。

### 4. 生成看不到结果；按钮颜色

- **结果**：已知风险包括 **远程 `Image()` 兼容性、本地 `file://` URI 在不同系统版本上的表现、以及历史截断导致的 URL 损坏**——当前主干已侧重 **下载落盘预览**；若仍失败，需在设备上对照 **日志 + 单个 URL curl** 做排查。  
- **按钮**：与编辑主按钮统一到**同一套渐变 Capsule**，你的直觉与设计系统原则一致。

### 5. 历史记录还行

- **认同**：结构与参考意图接近；后续可做「删除单行、清空、按类型筛选」，属增强项。

### 6. 设置页未动

- **认同**：从「完成作业闭环」角度，至少应 **占位 → 最小可用**：例如清空历史、显示/修改服务端地址（可选）、展示构建号等。
